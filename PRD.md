## 1. Описание продукта

### 1.1 Краткое описание

**«Board Platform»** — закрытая веб-платформа для цифровизации работы Наблюдательного совета (Совета директоров) акционерного общества. Система объединяет в единое пространство проведение заседаний, голосование, документооборот, мониторинг KPI, внутренний аудит, видеоконференции и коммуникацию между членами совета и руководителями компании.

### 1.2 Цель продукта

Обеспечить полный цикл корпоративного управления Наблюдательного совета в цифровом формате:
- Планирование и проведение заседаний
- Принятие решений через систему голосования
- Автоматизированное формирование протоколов (с использованием ИИ)
- Контроль исполнения решений и мониторинг KPI
- Безопасный обмен документами и их согласование
- Прозрачный аудит всех действий

### 1.3 Какую проблему решает

| Проблема | Решение платформы |
|----------|-------------------|
| Разрозненность каналов коммуникации (email, мессенджеры, бумажный документооборот) | Единая защищённая платформа для всех процессов |
| Непрозрачность принятия решений | Система голосования с фиксацией каждого голоса и аудит-логом |
| Сложность организации заседаний (особенно при удалённом участии) | Встроенные видеоконференции + календарь собраний |
| Долгое формирование протоколов заседаний | ИИ-генерация протоколов на основе брифа заседания |
| Отсутствие контроля исполнения решений | Мониторинг KPI с прогресс-барами и ответственными |
| Сложность согласования документов | Электронный документооборот с workflow согласования |
| Отсутствие единого хранилища корпоративной информации | Профили членов совета, правления, структура органов управления |

### 1.4 Основная ценность для пользователей

- **Для Председателя НС:** полный контроль над процессами, быстрое принятие решений, прозрачность
- **Для членов НС:** удобный доступ к материалам, голосование онлайн, видеоконференции
- **Для Правления:** оперативная коммуникация с НС, согласование документов, отчётность
- **Для корпоративного секретаря:** автоматизация протоколов, управление календарём, документооборот
- **Для аудитора:** полный аудит-трейл всех действий в системе

## 2. Целевая аудитория

### 2.1 Кто будет пользоваться системой

Платформа предназначена для замкнутого круга лиц — участников системы корпоративного управления акционерного общества:

| Группа | Описание |
|--------|----------|
| Члены Наблюдательного совета | 3–15 человек, принимающие стратегические решения |
| Члены Правления | Исполнительный орган компании, подотчётный НС |
| Руководители подразделений | Участвуют в чатах, предоставляют информацию |
| Корпоративный секретарь / Администратор | Обеспечивает организационную работу |
| Внутренние аудиторы | Проводят проверки и формируют отчёты |

### 2.2 Задачи пользователей

- Участвовать в заседаниях (очно или по видеосвязи)
- Голосовать по вопросам повестки дня
- Знакомиться с материалами к заседаниям и документами
- Согласовывать документы в электронном виде
- Контролировать исполнение решений через KPI
- Общаться с руководителями компании по рабочим вопросам
- Просматривать статистику и отчёты по деятельности совета
- Проводить внутренний аудит

### 2.3 Боли пользователей

- Необходимость физического присутствия для голосования
- Ручное ведение протоколов
- Рассылка документов по email без контроля версий
- Отсутствие единой платформы — приходится использовать 5–7 разных инструментов
- Непрозрачность статуса согласования документов
- Сложность отслеживания прогресса исполнения решений
- Невозможность оперативного созыва экстренного заседания

### 2.4 Роли пользователей (обзор)

1. **Председатель Наблюдательного совета** — полный доступ ко всем разделам
2. **Член Наблюдательного совета** — голосование, просмотр, участие в заседаниях
3. **Член Правления** — ограниченный доступ, отчётность, коммуникация
4. **Корпоративный секретарь (Администратор)** — управление системой, создание заседаний
5. **Аудитор** — доступ к аудит-модулю и статистике
6. **Руководитель подразделения** — чат, документооборот (ограниченно)

## 3. Полный список функций

### 3.1 Информация об Обществе

| Параметр | Описание |
|----------|----------|
| **Название** | Информация об Обществе |
| **Назначение** | Центральный справочник по органам управления компанией: состав НС, Правления и Членов Правления с профилями каждого участника |
| **Основной сценарий** | Пользователь переходит в раздел → видит сводку (количество членов НС, Правления, Управляющих) → переключается между вкладками «Наблюдательный совет», «Правление», «KПO Членов Правления» → просматривает карточку каждого участника (ФИО, должность, образование, опыт, контакты, срок полномочий) |
| **Ограничения** | Редактирование профилей доступно только Администратору. Контактная информация скрыта для ролей без соответствующего доступа |

### 3.2 Панель управления (Dashboard)

| Параметр | Описание |
|----------|----------|
| **Название** | Панель управления |
| **Назначение** | Главная страница после авторизации — обзор предстоящих собраний, активных голосований и быстрые действия |
| **Основной сценарий** | Пользователь входит в систему → видит: предстоящие собрания (список с датами и временем), активные голосования (требующие его участия), быстрые действия (начать видеоконференцию, создать протокол, написать руководителю, посмотреть статистику) → переходит к нужному разделу |
| **Ограничения** | Содержимое панели зависит от роли пользователя. Быстрые действия отображаются контекстно |

### 3.3 Календарь собраний

| Параметр | Описание |
|----------|----------|
| **Название** | Календарь собраний |
| **Назначение** | Визуальное отображение расписания заседаний НС в формате календаря |
| **Основной сценарий** | Пользователь переходит в «Календарь» → видит месячный или недельный вид → на ячейках дат отображены запланированные заседания → при клике на заседание открываются детали (повестка, участники, материалы) → можно перейти к видеоконференции или голосованию |
| **Ограничения** | Создание/редактирование заседаний — только для Администратора и Председателя. Обычные члены только просматривают. Кнопка «Сегодня» возвращает к текущей дате |

### 3.4 Система голосования

| Параметр | Описание |
|----------|----------|
| **Название** | Система голосования |
| **Назначение** | Онлайн-голосование по вопросам повестки заседаний с использованием квалифицированных правил (кворум, большинство) |
| **Основной сценарий** | Пользователь переходит в «Голосование» → видит список активных голосований → выбирает вопрос → читает описание и приложенные документы → голосует «За», «Против» или «Воздержался» → видит прогресс голосования (полоса с цветовой индикацией) → после завершения видит итоги |
| **Особенности** | Каждое голосование имеет: название, описание, срок, статус (активное/завершённое), счётчики голосов (За/Против/Воздержался/Не голосовали), прогресс-бар. Отображаются кнопки «Против», «Воздержаться», «Подробнее». Есть переключатель «Требуется кворум» |
| **Ограничения** | Голосовать могут только члены НС. Голос нельзя изменить после подачи (либо по решению Председателя). Голосование закрывается автоматически по истечении срока |

### 3.5 Протоколы собраний (с ИИ)

| Параметр | Описание |
|----------|----------|
| **Название** | Протоколы собраний |
| **Назначение** | Формирование, хранение и просмотр протоколов заседаний с возможностью ИИ-генерации |
| **Основной сценарий** | Пользователь переходит в «Протоколы» → выбирает собрание из выпадающего списка → нажимает «Получить бриф заседания» → просматривает бриф в панели предварительного просмотра → нажимает «Сформировать протокол с ИИ» → система генерирует протокол → пользователь редактирует и утверждает |
| **Особенности** | Список недавних протоколов с датами. Двухпанельный интерфейс: слева — создание, справа — предварительный просмотр. ИИ использует данные заседания (повестка, голосования, участники) для формирования протокола |
| **Ограничения** | Создание протоколов доступно Администратору и Председателю. Остальные — только просмотр утверждённых протоколов |

### 3.6 Чат с руководителями

| Параметр | Описание |
|----------|----------|
| **Название** | Чат с руководителями |
| **Назначение** | Защищённый мессенджер для коммуникации членов НС с руководителями подразделений |
| **Основной сценарий** | Пользователь переходит в «Чат» → видит список контактов (ФИО, должность, подразделение, индикатор онлайн) → выбирает собеседника → ведёт переписку в реальном времени → может искать контакты через поиск |
| **Особенности** | Каждый контакт отображает: аватар (инициалы), ФИО, должность, подразделение, онлайн-статус. Сообщения с временными метками. Поле ввода сообщения внизу |
| **Ограничения** | Чат только индивидуальный (1-на-1). Групповые чаты — возможное расширение. История сообщений сохраняется |

### 3.7 Документооборот

| Параметр | Описание |
|----------|----------|
| **Название** | Документооборот |
| **Назначение** | Управление документами и workflow согласования |
| **Основной сценарий** | Пользователь переходит в «Документооборот» → видит раздел «Требуется ваше согласование» (карточки документов с описанием, создателем, датой, ответственным, списком согласующих) → может скачать документ → нажимает «Согласовать» или «На согласовании» → ниже видит раздел «Мои документы» и «Согласованные документы» |
| **Особенности** | Карточка документа содержит: название, описание, создатель, дата создания, ответственный, список согласующих, статус (На согласовании / Согласован), кнопки «Скачать» и «Согласовать». Кнопка «Создать документ» для загрузки новых документов |
| **Ограничения** | Максимальный размер файла — ограничен (определить при разработке). Поддерживаемые форматы: PDF, DOCX, XLSX, PPTX. Согласование — необратимое действие |

### 3.8 Видеоконференция

| Параметр | Описание |
|----------|----------|
| **Название** | Видеоконференция |
| **Назначение** | Проведение онлайн-заседаний совета директоров |
| **Основной сценарий** | Пользователь переходит в «Видеоконференция» → видит приглашение «Готовы начать собрание?» → нажимает «Начать видеоконференцию» → подключается к конференции → ниже отображается список предстоящих собраний (название, дата, количество участников) |
| **Особенности** | Интеграция с внешним сервисом видеоконференций (или встроенный WebRTC). Список предстоящих собраний с датой и количеством участников |
| **Ограничения** | Требуется камера и микрофон. Максимальное количество участников — определить при разработке. Запись заседания — опциональная функция |

### 3.9 Статистика и отчёты

| Параметр | Описание |
|----------|----------|
| **Название** | Статистика и отчёты |
| **Назначение** | Аналитика деятельности органов корпоративного управления |
| **Основной сценарий** | Пользователь переходит в «Статистика» → выбирает орган управления и период → видит сводные карточки (всего собраний, создано/в работе, принято решений, среднее число участников) → ниже — таблица «История собраний» (дата, название, статус, формат, длительность, участники) → может экспортировать отчёт в PDF |
| **Особенности** | Фильтры: орган управления, период (последний квартал и т.д.), формат экспорта (PDF). Карточки KPI с иконками. Таблица истории собраний с сортировкой |
| **Ограничения** | Данные обновляются не в реальном времени (задержка зависит от архитектуры). Экспорт доступен Председателю и Администратору |

### 3.10 Мониторинг KPI (KПO Членов Правления)

| Параметр | Описание |
|----------|----------|
| **Название** | Мониторинг KPI |
| **Назначение** | Контроль выполнения ключевых показателей эффективности членами Правления |
| **Основной сценарий** | Пользователь переходит на вкладку «KПO Членов Правления» (в разделе «Информация об обществе») → видит общий мониторинг KPO (прогресс-бар, например 95%) → ниже — карточки отдельных KPI (название: «Выручка компании», ответственный, вкладки «Подтверж.» / «Ожидание» / «Сводка», план/факт/выполнение в процентах с цветовой шкалой) |
| **Особенности** | Цветовая индикация прогресса: зелёный (>80%), жёлтый (50–80%), красный (<50%). Каждый KPI привязан к ответственному. Три вкладки на каждом KPI: подтверждённые, ожидающие подтверждения, сводка |
| **Ограничения** | Ввод данных KPI — только Администратором или через интеграцию. Члены НС — только просмотр |

### 3.11 Внутренний аудит

| Параметр | Описание |
|----------|----------|
| **Название** | Внутренний аудит |
| **Назначение** | Контроль и анализ деятельности компании через аудиторские проверки |
| **Основной сценарий** | Пользователь переходит в «Внутренний аудит» → видит сводные карточки (активные проверки, завершённые, количество замечаний) → может выгрузить отчёт (PDF) → ниже — таблица «Активные проверки» (название, департамент, статус с цветовой индикацией: средний/высокий/критический, уровень риска, дата начала, дата окончания, аудитор) → ещё ниже — «Детализация замечаний» |
| **Особенности** | Статус проверки отображается цветными бейджами. Уровни приоритета: Средний (жёлтый), Высокий (оранжевый), Критический (красный). Экспорт отчёта. Привязка к департаменту и аудитору |
| **Ограничения** | Раздел доступен только Председателю, Аудитору и Администратору. Создание проверок — только Аудитором или Администратором |

### 3.12 Общее собрание акционеров

| Параметр | Описание |
|----------|----------|
| **Название** | Общее собрание акционеров |
| **Назначение** | Управление и проведение общих собраний акционеров |
| **Основной сценарий** | Пользователь переходит в «Общее собрание акционеров» → видит вкладки «Предстоящие собрания» и «Прошедшие собрания» → выбирает собрание → видит детали: тип (Годовое / Внеочередное), дата и место, статус, повестка дня (нумерованный список вопросов), материалы (прикреплённые документы: Финансовый отчёт, Адресное заявление, Отчёт НС и т.д.) → может просмотреть материалы |
| **Особенности** | Вкладки: предстоящие / прошедшие. Карточка собрания: тип, дата, место, кворум. Повестка дня с нумерацией. Секция материалов с карточками документов (цветные иконки) |
| **Ограничения** | Создание собраний — только Администратор. Материалы загружает Администратор или Председатель |

## 4. Структура страниц / экранов

### 4.1 Страница авторизации (Login)

| Параметр | Описание |
|----------|----------|
| **Назначение** | Вход в закрытую систему |
| **Элементы** | Логотип, поля email и пароль, кнопка «Войти», ссылка «Забыли пароль?» |
| **Действия** | Ввод учётных данных, вход в систему |
| **Данные** | Запрос к auth-сервису Supabase |
| **Состояния** | Пустая форма, загрузка, ошибка авторизации, успешный вход (редирект) |
| **Роли** | Доступна всем. После входа — редирект на панель управления с учётом роли |

### 6.3 Принципы разграничения доступа (RLS)

Supabase Row Level Security (RLS) включена на всех таблицах. Основные политики:

| Таблица | Политика | Описание |
|---------|----------|----------|
| `users` | SELECT: все авторизованные | Все могут видеть базовую информацию. Контакты — только для ролей chairman, admin, board_member |
| `meetings` | SELECT: все авторизованные | INSERT/UPDATE: только chairman, admin |
| `votings` | SELECT: все авторизованные | INSERT: chairman, admin. Голосование: board_member, chairman |
| `votes` | SELECT: только свой голос + chairman/admin видят все | INSERT: board_member, chairman (только свой) |
| `protocols` | SELECT: все авторизованные | INSERT/UPDATE: chairman, admin |
| `messages` | SELECT: только участники переписки | INSERT: отправитель = текущий пользователь |
| `documents` | SELECT: автор + согласующие + admin + chairman | INSERT: все авторизованные |
| `document_approvals` | SELECT: участники согласования | UPDATE: только назначенный согласующий |
| `kpi_records` | SELECT: chairman, admin, board_member, ответственный | UPDATE: admin |
| `audits` | SELECT: chairman, admin, auditor | INSERT/UPDATE: auditor, admin |
| `audit_log` | SELECT: chairman, admin | INSERT: система (через серверную функцию) |

### 7.1 Стек технологий

| Компонент | Технология | Обоснование |

| **ИИ (протоколы)** | OpenAI API / Anthropic API | Генерация протоколов из брифов |
| **Хостинг** | beget.com (VPS) | Требование заказчика |
| **Контроль версий** | Git | Стандарт |

## 8. Нефункциональные требования

### 8.1 Безопасность

| Требование | Реализация |
|------------|------------|
| Аутентификация | Supabase Auth (JWT-токены) |
| Авторизация | RLS-политики на уровне БД + middleware на сервере |
| Шифрование в передаче | HTTPS (TLS 1.3) обязательно для всех соединений |
| Шифрование в хранении | Supabase Storage encryption at rest |
| Защита от CSRF | CSRF-токены на формах |
| Защита от XSS | Sanitization входных данных, CSP-заголовки |
| Защита от SQL-инъекций | Параметризованные запросы (Supabase SDK) |
| Защита от брутфорса | Rate limiting на auth-эндпоинтах |
| Сессии | JWT с коротким TTL (15 мин access, 7 дней refresh) |
| Пароли | Минимум 8 символов, bcrypt-хеширование (Supabase Auth) |
| IP-фильтрация | Опционально — ограничение доступа по IP (whitelist) |

### 8.2 Производительность

| Метрика | Целевое значение |
|---------|------------------|
| Время загрузки страницы (First Contentful Paint) | < 1.5 сек |
| Время ответа API | < 300 мс (p95) |
| Время загрузки списка заседаний | < 500 мс |
| Максимальное количество одновременных пользователей | 50 |
| Время доставки сообщения (чат) | < 500 мс |
| Размер бандла фронтенда (gzip) | < 500 KB |

### 8.3 Масштабируемость

- Горизонтальное масштабирование Node.js через PM2 (cluster mode)
- Supabase Pro-план при росте нагрузки
- CDN для статических ресурсов (при необходимости)
- Пагинация на всех списках (по 20 элементов)
- Ленивая загрузка компонентов (React.lazy)

### 8.4 Защита данных

- Все данные хранятся на территории, соответствующей требованиям регулятора
- Персональные данные (ФИО, email, телефон) обрабатываются в соответствии с локальным законодательством
- Документы доступны только авторизованным пользователям через signed URLs
- Удалённые данные помечаются soft-delete (поле `is_deleted`), физическое удаление — через отдельную процедуру

### 8.5 Резервное копирование

| Параметр | Значение |
|----------|----------|
| Частота бэкапов БД | Ежедневно (автоматически Supabase) |
| Хранение бэкапов | 30 дней |
| Бэкап файлового хранилища | Еженедельно |
| Восстановление (RTO) | < 4 часа |
| Потеря данных (RPO) | < 24 часа |

### 8.6 Логирование

| Уровень | Что логируется |
|---------|---------------|
| **Application** | Ошибки (error), предупреждения (warn), информация (info) — через Winston/Pino |
| **Access** | HTTP-запросы (метод, URL, статус, время ответа) — через Morgan |
| **Audit** | Все бизнес-действия пользователей — в таблицу `audit_log` |
| **Security** | Попытки входа, неудачные авторизации, подозрительная активность |
| **Хранение логов** | Файлы на сервере + ротация (30 дней) |
